# 协议状态机说明

## 发射机状态 (Transmitter)

| 状态 | 枚举值 | 含义 |
|------|--------|------|
| **INIT** | 0 | 处理文件、FEC编码、预调制所有数据帧和跳频时隙 |
| **WAIT_READY** | 1 | 在 2.5GHz 锚点频率上周期性发送 BEACON，同时监听 1.45GHz 反馈通道，等待接收机回复 ACK |
| **START_COUNTDOWN** | 2 | 收到 BEACON 的 ACK 后，持续发送 START 帧并通过 ACK 确认接收机已就绪。收到 ACK 且已发 ≥5 次 START 后进入 DATA_ONCE。超过 50 次无 ACK 则退回 WAIT_READY |
| **DATA_ONCE** | 3 | 按跳频序列依次发送所有数据时隙，每个时隙在不同频率上，发完即走，不重传 |
| **END_LISTEN** | 4 | 数据发完后，在锚点频率重复发送 END 帧，同时监听反馈通道等待接收机的 RESULT 报告 |
| **DONE** | 5 | 收到 RESULT 或超时，关闭 SDR 资源，退出 |

---

## 接收机状态 (Receiver)

| 状态 | 枚举值 | 含义 |
|------|--------|------|
| **WAIT_BEACON** | 0 | 在 2.5GHz 锚点频率监听，等待发射机的 BEACON 帧。同时周期性发送 READY 发现信号。检测到 BEACON 后构建正确 session_id 的 ACK 并回复 |
| **READY_SENT** | 1 | ACK 已发出，继续监听锚点频率等待 START 帧（包含跳频参数）。收到 START 后**立即回 ACK 确认**，然后进入 FOLLOW_HOP |
| **FOLLOW_HOP** | 2 | 按 START 中指定的跳频序列，逐时隙切换频率接收数据。仅在成功收到数据后才推进 slot_ptr。若 3 秒无数据，重发 ACK 与 TX 重新同步 |
| **FEC_REBUILD** | 3 | 所有时隙接收完毕，用 RaptorQ FEC 从已收到的帧中重建原始文件 |
| **RESULT_REPORT** | 4 | 在反馈频率上重复发送 RESULT 帧，告知发射机接收结果（恢复率、CRC 是否匹配等） |
| **DONE** | 5 | 关闭 SDR 资源，退出 |

---

## 协议流程图

```
                    BEACON (周期性)
TX:  INIT ──→ WAIT_READY ──────────────┐
                                        ↓
                   ACK (session_id匹配)  发START (持续)
TX:           WAIT_READY ───────────→ START_COUNTDOWN ────────────┐
                                                                ↓
                   ACK (RX确认收到START, 且TX已发≥5次START)    DATA_ONCE
TX:           START_COUNTDOWN ──────────────────────────────→ DATA_ONCE
                                                           │
                                                    DATA 时隙 (跳频)
TX:           DATA_ONCE ─────────────────────────────────────────→ END_LISTEN ──→ DONE
                                                                       │
                                                                  END 帧 / RESULT
                                                                       │
                                                                       ↓
RX:  WAIT_BEACON ──→ READY_SENT ──→ FOLLOW_HOP ──→ FEC_REBUILD ──→ RESULT_REPORT ──→ DONE
       │    ↑             │               │                                   │
       │ 检测BEACON    收到START     收到数据推进slot_ptr                 RESULT帧
       │ 回ACK        立即回ACK      无数据3秒→重发ACK
       │              进入FOLLOW_HOP  无数据→不推进slot_ptr
       │
       └── 周期性发READY发现信号
```

### START 握手确认机制（关键改进）

```
TX START_COUNTDOWN:                     RX READY_SENT/FOLLOW_HOP:
                                          
  start_count = 0                       收到 START 后:
    ↓                                     ├─ 立即发 ACK (不等周期)
  每次循环:                               ├─ 进入 FOLLOW_HOP
    ├─ 发 START (锚点 2.5GHz)              └─ slot_ptr=1, last_sync_time=tic
    ├─ 监听反馈 (1.45GHz)              
    ├─ start_count++                    FOLLOW_HOP 中:
    │                                     ├─ 收到数据 → 解码 → slot_ptr++
    ├─ 收到 ACK && start_count >= 5?      │              → last_sync_time=tic
    │   └─ YES → 进入 DATA_ONCE           │
    │                                     └─ 3秒无数据 → 发 ACK
    └─ start_count > 50 (超时)?              → last_sync_time=tic (重置计时)
        └─ YES → 退回 WAIT_READY
```

### 关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| MIN_START_SENDS | 5 | TX 最少发 5 次 START 才接受 ACK 进入数据阶段 |
| MAX_START_ATTEMPTS | 50 | TX 超时 (~30s) 后退回 WAIT_READY |
| RX_DATA_TIMEOUT | 3 秒 | RX 无数据时重发 ACK 的间隔 |

## 握手帧类型

| 帧类型 | 枚举值 | 方向 | 说明 |
|--------|--------|------|------|
| **BEACON** | 100 | TX → RX | 握手信标，携带会话ID |
| **ACK** | 101 | RX → TX | 握手确认（BEACON确认、START确认、保活、超时重同步） |
| **READY** | 30 | RX → TX | 发现信号，RX周期性发送用于探测 |
| **START** | 41 | TX → RX | 启动帧，携带跳频种子、时隙总数、码字/时隙、FEC参数 |
| **END** | 42 | TX → RX | 结束帧，数据发送完成 |
| **RESULT** | 32 | RX → TX | 结果帧，报告接收恢复状态 |

## 常见问题排查

### 握手失败（RX 一直在 WAIT_BEACON）

1. RX SamplesPerFrame 是否 ≥ 80,000（BEACON 波形约 43,000 采样点）
2. TX BEACON 是否全幅度发送（非 0.1x 衰减）
3. decode_ctrl_hs 的 Threshold 是否为 250
4. 两端 USRP 时钟是否同步（ClockSource='External'）

### 握手成功但 RX 收不到 START

1. TX 是否正确进入 START_COUNTDOWN（收到 ACK 后）
2. RX 收到 BEACON 后是否正确构建了 ACK（session_id 匹配）
3. start_count >= MIN_START_SENDS (5) 时 TX 是否收到 RX 的 ACK

### 数据时隙全部"无同步"

1. TX 数据缩放是否足够（当前 0.8x）
2. RX 检测阈值是否合适（当前 180）
3. 跳频序列是否一致（hop_seed、num_carriers 相同）
4. 两端 link_phy_defs.m 中 head_data 是否一致
5. detect_hop_slot 的 rxfilter 是否复用（避免每次重建）

### RX 在 FOLLOW_HOP 中卡住

1. 查看是否有 "3秒无数据, 重发ACK" 日志 — 正常行为，RX 在等 TX
2. 查看 [RX-DETECT] 诊断输出中的最大相关值
3. 确认 TX 是否已进入 DATA_ONCE

### 程序崩溃

1. rx_frame_cache_update 是否正确初始化了 received_map 字段
2. 查看 [RX-DETECT] 诊断输出中的最大相关值，判断信号强度
